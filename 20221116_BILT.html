Current state of BIM and how to improve it

"Our industry works in a document-centric manner rather than a data-centric one"

- Exchange of files rather than data integration
- A bim is one database - we need multiple
- To do a heat loss calculation, information that is stored in architectural drawings is required
- Thermal requirements, room usage, areas, lengths, room IDs
- The connections is broken because we use closed systems to store the information
- This is only a small part of the big process of developing the MEP design of a building
- There are also other disciplines that have an influence on the heat loss calculation
- And they store the information again in closed file formats
- Here you see the big process of developing the MEP design of a building
- Characterized by closed file formats with broken connections, where there should be a direct relation between them

"Exchange of BIM-data as files makes data integration impossible"

- BCF as an example
- You cannot exchange issues in a file. How about the ones you resolve in the mean time?

"We need to switch from Today's monolitic nature of BIM to a network based one"

- Fine that the different BIM systems rely on an closed data model, but they should be able to exchange resulting geometry and derived properties

So how do we do this?

The idea is to use the existing infrastructure of the World Wide Web to mediate a distributed, network-based BIM

Linked Data

The concept of Linked Data was coined in 2006 by sir Tim Berners-Lee, the inventor of the web, and is therfore not entirely new üë¥üëµ

It basically refers to a set of best practices for publishing structured data on the Web

Linked Data Principles

  1. Use URIs (Uniform Resource Identifiers) as names for things
  2. Use HTTP URIs (web addresses) so that people can look up those names üëÄ
  3. When someone looks up a URI, provide useful information, using the standards (RDF, SPARQL) (more on this later) üí°
  4. Include links to other URIs so that they can discover more things üî≠üß≠üó∫Ô∏è

Data is stored and hosted by the stakeholder who generated it and is responsible for it

Data can be shared with authenticated people or be made publicly available

The model data can be extended with Linked Open Data (product- and material properties, IoT, GIS)

Even the schemas are distributed, and can therefore extend one another

Current situation: Open standards in isolated closed worlds

Linked data benefit: Standards use, extend and refer to one another

RDF vs. tables

Most of us are used to storing information in spreadsheets (or relational databases (RDBs) for the more advanced), but in RDF we use a graph structure.

RDF: Resource Description Framework

(Alex) Linked Data is using a graph structure, based on the Resource Description Framework (RDF), creating an interconnected knowledge graph that is much more powerful than storing data in conventional tables. Knowledge graphs consist of relationships between things, stored as triples.
(Alex) A triple always consists of three things - A subject, predicate and an opject.

(Alex) In the context of a building, if we want to declare a relationship between a wall and a window, we can say:

Wall_002 has window Window_022

But how do we get from tables to graphs?

(Alex) From an RDF perspective we see too subjects. One for each data row. And they are even identified with an ID.
(Alex) Each column contains a statement about those subjects and the predicate value is given by the column header.
(Alex) The value of the subject-predicate pair can then be found in the cell.
(Alex) So in RDF perspective the table can be written in raw data like this.
And as a graph, we can visualize the data like this.
(Alex)
- Combining data stored in different tables into one unified representation is possible if there is a common identifier for the rows.
- For example, we could have another table with middle names.
- It's not a problem that there is no middle name for Alexander. RDF builds on an open world assumption meaning that if we have the information that's great and if we don't that's also alright. Then Alexanders middle name is simply unknown to us.
(Alex) In RDBs it's common to have dedicated tables that deal with many-to-many relationships. These are also easily translated.

Compared to RDBs, RDF makes sense from a logical perspective and not only from a data management perspective.

This benefit will be more obvious when we start to query the datasets!

Linked Building Data (LBD) Community Group

Established December 2014
195 members November 2022

A Community to gather best practice use cases in the field of BIM and Linked Data technologies

  • Regular meetings to share and discuss new use cases
  • Develop and share external ontologies within Linked Building Data
  • Current ontologies include OMG (Ontology for Managing Geometry), BOT (Core topological concepts of a building), BPO (Building Product Ontology) and many more
  • Everyone is invited to build on top of this foundation, extend the existing ontologies and share new ontologies to the group!

Some practical examples

BART Stage 2 delivery

(Alex) In this small project, the intention was to check a building model upon fire damper requirements.
- If a pipe is going through a wall with fire classification (wall between two different fire sections), a fire damper has to be installed.
- If the fire zoning changes, the requirement of fire damper changes as well. This is conidered here by using incremental reasoning with the graph database solution RDFox from Oxford Semantics.
- We can write a rule: If two spaces from different fire sections share the same wall interface, this wall interface is classified as a fire protection interface
- If a pipe is now intersecting with such a fire classified interface, it must contain a fire damper.

Incremential reasoning

(Alex)
1. Enter fire sectioning into excel file and visualize in 3D viewer
2. Show fire walls based on the fire sectioning
3. Check if the required fire dampers are installed
4. Change fire sectioning in excel file, upload and see what changed
